# 상품 도메인 정책 명세서

## 1. 상품 조회 정책

1. 상품은 `status = ACTIVE` 인 경우에만 조회 가능하다.
2. `INACTIVE`, `DELETED` 상태의 상품은 API 조회 대상에서 제외한다.
3. 상품 상세 조회 시 아래 정보를 함께 조회한다.

    * 상품 기본 정보 (`product`)
    * 옵션 그룹 (`option_group`)
    * 옵션 값 (`option_value`)
    * SKU (`sku`)
    * SKU 옵션 매핑 (`sku_option`)

---

## 2. 옵션 정책

1. 상품 옵션은 `option_group` 단위로 관리된다.
   예: Color, Size

2. `option_group` 하위에는 여러 `option_value`가 존재한다.
   예:

    * Color → Black, Red
    * Size → M, L

3. 실제 구매 가능한 옵션 조합은 `SKU`로 관리된다.

4. `SKU`는 하나 이상의 `option_value` 조합으로 구성된다.

예시

```
Color: Black
Size: M
→ SKU 존재
```

```
Color: Black
Size: XL
→ SKU 없음 → 선택 불가
```

---

## 3. SKU 정책

1. SKU는 실제 판매 단위이다.
2. SKU는 다음 정보를 가진다.

* SKU 코드 (`sku_code`)
* 가격 (`price`)
* 재고 (`stock`)
* 상태 (`status`)

3. 상품 옵션 선택 시 실제 구매 가능 여부는 SKU 기준으로 판단한다.

---

## 4. 재고 정책

1. 재고는 SKU 단위로 관리한다.
2. SKU 재고(`stock`)가 `0`이면 구매 불가이다.
3. 주문 생성 시 SKU 재고를 차감한다.
4. 재고 변경 이력은 `stock_history` 테이블에 기록한다.

재고 변경 사유 예시

```
ORDER
CANCEL
ADJUST
```

---

## 5. 가격 정책

1. 상품 기본 가격은 `product.base_price` 이다.
2. 실제 판매 가격은 `sku.price` 기준으로 결정된다.
3. 주문 생성 시점의 가격은 `order_item.price_snapshot`에 저장한다.

이유

```
상품 가격 변경 시
과거 주문 데이터 보존
```

---

## 6. 옵션 스냅샷 정책

1. 주문 생성 시 SKU 옵션 정보를 `order_item.option_snapshot`에 저장한다.
2. 상품 옵션 변경 시에도 과거 주문 데이터는 영향을 받지 않는다.

예시

```
주문 시점 옵션
Color: Black
Size: M
```

```
이후 옵션 변경
Color: Blue 추가
```

→ 기존 주문 데이터 유지

---

## 7. 상품 상태 정책

상품 상태 종류

```
ACTIVE      판매중
INACTIVE    판매중지
DELETED     삭제
```

정책

1. `ACTIVE` 상품만 사용자 조회 가능
2. `INACTIVE` 상품은 관리자만 조회 가능
3. `DELETED` 상품은 조회 불가

---

## 8. 상품 상세 조회 정책

상품 상세 조회 API는 다음 구조로 응답한다.

```
상품
 ├ 옵션 그룹
 │   └ 옵션 값
 │
 └ SKU 목록
      └ 옵션 조합
```

예시

```
상품: 스키니 바지

옵션
Color
 ├ Black
 └ Red

Size
 ├ M
 └ L

SKU
Black M
Black L
Red M
Red L
```

---

## 9. 상품 조회 성능 정책

상품 상세 조회 시 다음 컬럼에 인덱스를 적용한다.

```
option_group(product_id)
option_value(option_group_id)
sku(product_id)
sku_option(sku_id)
```

목적

```
상품 상세 조회 성능 최적화
```
